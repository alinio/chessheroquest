# ChessHeroQuest — Chess Curation Spec

**Purpose.** This is the single source of truth for every piece of *real chess content* the game needs: the DNA test positions, the Learn/Drill opening lines, the boss gauntlets, and the authored RPG explanation text. It defines the exact data shape, how to classify and verify each item, and which module consumes it.

**The non-negotiable rule.** All chess content is REAL and VERIFIED — real openings, real ECO/PGN lines, candidate moves tagged by the engine (Stockfish). All explanation prose is AUTHORED ONCE and stored here. **Nothing chess-related is ever generated by an LLM at runtime.** If an item isn't curated yet, it stays an explicit `TODO` — we never invent moves or fake a line to fill a gap.

## 3. Schema A — Test position (PRIORITY)

This is the shape for every position in the DNA test bank. One object per position.

```ts
type ArchetypeId = 'warrior' | 'strategist' | 'defender' | 'trickster';

interface ExplorerStats {       // snapshotted from the Lichess explorer at curation time — see §5b
  popularityPct?: number;       // % of games in this rating band that play this move
  mastersPct?: number;          // % in the Masters database
  eval?: string;                // engine eval as a label, e.g. "+0.3" | "=" | "-0.2"
  peerResults?: { whitePct: number; drawPct: number; blackPct: number }; // W/D/L among peers
  gamesCount?: number;          // sample size behind the percentages
  ratingBand?: string;          // e.g. "1400-1800" — the peer band these stats are for
  snapshotDate?: string;        // ISO date the stats were pulled
}

interface TestOption {
  san: string;             // the move in Standard Algebraic Notation, e.g. "Bf5"
  isBest?: boolean;        // SKILL questions only: exactly one option true
  centipawnLoss?: number;  // engine eval loss vs best (0 = best). SKILL only.
  optionNote: string;      // authored: what this move means / why it's good or bad
  archetypeLean?: ArchetypeId; // STYLE forks only: which archetype this choice signals
  explorer?: ExplorerStats;    // POST-ANSWER reveal only — NEVER shown before the player answers (§5b)
}

interface TestPosition {
  id: string;                 // stable, e.g. "carokann-b15-01"
  openingName: string;        // "Caro-Kann Defense"
  openingFamilyEco: string;   // "B10–B19"
  eco: string;                // specific tabiya ECO, e.g. "B15"
  fen: string;                // the position to solve — MUST be legal & match lineSan
  lineSan: string[];          // real moves from the start that reach this FEN
  sideToMove: 'white' | 'black';
  difficulty: 1 | 2 | 3 | 4 | 5;
  questionType: 'skill' | 'style';
  prompt?: string;            // optional override; else derived from questionType
  options: TestOption[];      // 2–4 options
  contextRpg: string;         // PRE-answer: orientation only, NO answer hint
  explanationRpg: string;     // POST-answer: the rich teaching reveal
  source: string;             // where it was verified (reference + engine depth)
  verified: boolean;          // true only after a human/engine check
}
```

## 4. Skill question vs style fork — how to classify

Decide this per position before writing anything else; it changes the framing, the scoring, and the context copy.

**SKILL question** — there is one clearly best move (a tactic, a key theoretical move, a punish).
- `questionType: 'skill'`, `prompt` defaults to "Best move?".
- Exactly one option has `isBest: true`; others carry a real `centipawnLoss`.
- Scores toward Opening IQ. Show correct/incorrect after answering.
- `contextRpg` may name the opening + show the line (orientation, never the idea/answer).

**STYLE fork** — two or more moves are all theory-approved; the choice reveals temperament, not skill.
- `questionType: 'style'`, `prompt` is neutral ("Your move?" / "How do you continue?"). **Never "Best move?".**
- No `isBest`. No correct/incorrect. **Must not penalize Opening IQ.**
- Each option may carry an `archetypeLean` as a secondary style signal (keep isolated from IQ).
- Hide `openingName` in `contextRpg`; reveal it in `explanationRpg`.

Rule of thumb: if a strong engine has the top 2 candidate moves within ~0.3 and both are named theory, it's a **style fork**. If there's a clear gap, it's a **skill** question.

---

## 5. RPG text guidelines

- **Two lengths.** `contextRpg` = 1–2 sentences, orientation only (line + whose move; opening name only for skill questions). `explanationRpg` = 2–4 sentences, the teaching reveal (what the opening is, the idea behind the best move / what each fork means, any trap).
- **Voice.** Epic but clear — light fantasy flavour over real chess substance. The chess must be correct first; the flavour is seasoning, never a substitute.
- **No spoilers pre-answer.** `contextRpg` must never hint at which move to pick.
- **Reusable per opening.** The "what this opening is" sentence can be shared across the test, the Learn intro, and the boss intro — write it once per opening and reuse.
- **Authored, never LLM at runtime.** Same rule as the moves.

## 5b. Explorer data layer (popularity / eval / peer results)

This is the Chessbook-style stat layer — popularity %, Masters %, engine eval, and peer win/draw/loss — attached to each candidate move. It is what makes a move feel *situated* ("82% of players choose this; the engine is fine with it"), and it is fully on-truth: it comes from the **Lichess opening explorer + Stockfish**, never from an LLM.

**Source & determinism.**
- Pull each move's stats from the Lichess opening explorer (Masters DB for `mastersPct`; the rated DB filtered to a rating band for `popularityPct` + `peerResults`) and the engine for `eval`.
- **Snapshot the values into the data at curation time** (store `gamesCount`, `ratingBand`, `snapshotDate`). Do NOT fetch live during the DNA test — the test must be deterministic and offline-safe. Learn/Drill may optionally refresh snapshots periodically (a curation job), not per-render.

**Where it is consumed.**
- **Learn / Drill (M6) — full treatment.** Show the candidate moves as a list with name + `popularityPct` + `eval` + the one-line `explain`, plus the move breadcrumb and the variation name updating per ply. This is the explorer-style presentation the founder liked — reskinned to our look.
- **DNA Test (M2) — POST-ANSWER only.** Reveal `explorer` as part of the after-answer explanation ("58% of players here choose this · engine ="). **Never before the player answers** — pre-answer stats would coach the player and inflate the Opening IQ.

**Peer-band personalization.** Filter `peerResults` to the player's own rating band so it reads "how players at *your* level fare," not a generic average. Store the band you pulled in `ratingBand`.

**Reskin, don't clone.** Render these as our gold-on-obsidian styled bars and labels (not the green Chessbook UI), using our terminology.

One tree per opening. Feeds the M6 interactive board and generates SRS drill cards from its nodes.

```ts
interface LineNode {
  san: string;
  fen: string;          // position AFTER this move
  explain?: string;     // authored note for this move (Learn captions)
  isCritical?: boolean; // becomes a drill card / a "must-know" move
  explorer?: ExplorerStats; // popularity / eval / peer results for this move (§5b) — full treatment in Learn
}

interface OpeningLineTree {
  openingId: string;        // matches the roster + test position ids
  openingName: string;
  eco: string;
  realm: string;            // from opening-boss-catalog.md
  accent: string;           // realm accent token
  mainLine: LineNode[];     // the backbone Black/White should know
  variations: { name: string; fromPly: number; line: LineNode[]; explain: string }[];
  traps: { name: string; line: LineNode[]; explain: string; refutation: LineNode[] }[];
}
```

## 7. Schema C — Boss gauntlet (M7, Phase 2)

One per Opening Guardian (per opening) and per Kingdom Boss (per realm).

```ts
interface BossGauntlet {
  bossId: string;
  bossName: string;          // e.g. "Ignar, the Crowned Conflagration"
  kind: 'opening-guardian' | 'kingdom-boss';
  openingId?: string;        // guardians only
  realm: string;
  format: string;            // how the fight is structured (from catalog)
  lines: { name: string; line: LineNode[]; testsWhat: string }[]; // the critical lines the boss plays/tests
  sealReward: string;        // the seal earned on a Medium win
}
```

> Win condition is beating the boss at **Medium** (validation) — keep `lines` to the genuinely critical, must-know continuations of that opening.

## 8. How to curate ONE position (the recipe)

1. **Pick** an opening + a meaningful moment in it (an early tabiya, a key decision, a known trap).
2. **Get the real line** from a trusted reference (opening explorer / a reputable repertoire). Copy the SAN moves into `lineSan`.
3. **Set the FEN** for the position to solve, and confirm it matches `lineSan` (play the moves out; the board must reproduce the FEN exactly).
4. **Tag the candidate moves with the engine** (Stockfish): record `centipawnLoss` for each option vs the best. This is what decides skill vs style (§4). While here, **snapshot the explorer stats** (popularity %, Masters %, peer W/D/L for the target rating band) into each option's / node's `explorer` (§5b).
5. **Classify** skill or style; set `prompt`, `isBest`/`archetypeLean` accordingly.
6. **Write** `contextRpg` (no spoiler) and `explanationRpg` (the reveal), per §5.
7. **Fill `source`** (reference + engine depth) and set `verified: true` only after the FEN/line/eval check passes.

We can wire Stockfish to auto-tag candidate moves so step 4 is mechanical — you supply the opening + line, the engine returns the evals, you write the prose.
